# Product

## Register

product

## Users

AI Review Copilot is for developers, QA engineers, and small product teams who need to review pull request diffs, turn QA context into structured bug reports, and produce PR-ready Markdown without sending sensitive source code to hosted AI services. Users are often working inside an active debugging or review session, under time pressure, with a high need for clarity, privacy, and repeatable output.

## Product Purpose

The product provides a local-first review workspace for code review and bug report workflows. It helps users paste or generate diffs, add ticket and QA context, run static checks or local Ollama analysis, and copy or export clean Markdown. Success means the interface feels fast, trustworthy, practical, and private while keeping the user's attention on the review task instead of the tool.

## Brand Personality

Precise, local-first, calm, professional, and developer-focused. The product should feel trustworthy, practical, private, and efficient. It should not feel flashy, overly corporate, or like a generic AI demo.

## Anti-references

Avoid generic SaaS marketing gloss, over-decorated AI dashboards, noisy animated UI, excessive gradients or glows, random blue AI aesthetics, bloated cards, and anything that makes the workflow feel slower or less serious.

## Design Principles

- Keep the workflow first: every visual choice should make review, QA context entry, analysis, and Markdown export easier to scan and complete.
- Earn trust through restraint: use polished hierarchy, clear labels, and predictable controls rather than decorative spectacle.
- Preserve local-first confidence: make browser-to-Ollama behavior, privacy boundaries, and static fallback options understandable without adding friction.
- Support long work sessions: favor readable contrast, stable layouts, calm motion, and comfortable density over dramatic presentation.
- Make advanced work feel practical: expose power through sensible controls, examples, and status feedback instead of hidden magic.

## Accessibility & Inclusion

Use WCAG AA as the baseline. The UI should be keyboard-friendly, reduced-motion aware, readable, high-contrast, and usable for developers and QA teams working for long periods. Avoid low-contrast muted text, motion that does not convey state, and layouts that depend on color alone.
